# Diocesan Finance API - Modifications Guide

This document explains the modifications made to the original system: three-role architecture, priest token system, collections CRUD, and debtors dashboard.

---

## 1. Three-Role Architecture

### User Roles

| Role | Permissions | Login Method | Use Case |
|------|---|---|---|
| **ADMIN** | Full access - create parishes, manage collections, upload data, generate priest tokens, view all dashboards | Email + Password | Diocese finance officer |
| **BISHOP** | Read-only dashboard and individual parishes, cannot edit or upload | Email + Password | Bishop overseeing diocese |
| **PRIEST** | Read-only access to own parish only, identified by static token | Priest Token (generated by admin) | Parish priest |

### Database Schema Changes

```sql
-- Updated users table
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  email VARCHAR(255),           -- NOT required for PRIEST
  password_hash VARCHAR(255),   -- NOT required for PRIEST
  priest_token VARCHAR(255) UNIQUE,  -- ONLY for PRIEST
  token_generated_by INTEGER,   -- Which ADMIN generated this token
  first_name VARCHAR(100),
  last_name VARCHAR(100),
  role VARCHAR(50) CHECK (role IN ('ADMIN', 'BISHOP', 'PRIEST')),
  parish_id INTEGER,
  is_active BOOLEAN DEFAULT true,
  token_expires_at TIMESTAMP,   -- Future: for token expiration
  created_at TIMESTAMP,
  updated_at TIMESTAMP
);
```

## 2. Priest Token System

### How It Works

1. **Admin creates priest user** (with name, parish assigned, NO email/password)
2. **Admin generates a unique static token** (32-character hex string)
3. **Admin gives token to priest** (printed, emailed, or verbal)
4. **Priest uses token to login** (instead of email/password)
5. **If priest rotates**, admin generates new token
6. **Old token becomes invalid immediately**

### GraphQL Mutations

#### Generate Token (First Time)
```graphql
mutation {
  generatePriestToken(priestUserId: 5) {
    id
    firstName
    lastName
    priestToken      # Securely display to admin
    role
  }
}
```

#### Rotate Token (Priest Reassignment)
```graphql
mutation {
  rotatePriestToken(priestUserId: 5) {
    id
    priestToken      # New token for new priest
    updatedAt
  }
}
```

#### Revoke Token (Deactivation)
```graphql
mutation {
  revokePriestToken(priestUserId: 5) {
    id
    priestToken      # Will be NULL
    isActive
  }
}
```

### Priest Login Flow

Priests **do NOT login with email/password**. Instead:

```graphql
mutation {
  loginWithPriestToken(token: "abc123def456...") {
    token           # JWT for subsequent API calls
    refreshToken
    user {
      id
      firstName
      lastName
      parish {
        name
      }
    }
  }
}
```

The returned `token` is a JWT that expires in 24 hours. Priests use this JWT for all subsequent GraphQL queries.

### Authorization by Role

```typescript
// ADMIN → Full access
requireRole(user, ['ADMIN'])

// BISHOP → Read-only views
requireRole(user, ['ADMIN', 'BISHOP'])

// PRIEST → Own parish only
if (user.role === 'PRIEST' && user.parish_id !== parishId) {
  throw new Error('Unauthorized');
}
```

## 3. Collections Management

### Before: Static Collections
Previously, 7 collections were pre-defined:
- First Collection
- Second Collection
- Tithe
- Harvest
- Special Offering
- Donations
- Other Income

### Now: Dynamic Collections (Admin CRUD)

Admins can now:
- **Create** new collections
- **Update** collection names/descriptions
- **Delete** (soft-delete) collections
- **List** all active collections

### GraphQL Mutations

#### Create Collection
```graphql
mutation {
  createCollection(
    name: "Christmas Offering"
    description: "Annual Christmas collection"
  ) {
    id
    name
    description
    createdBy {
      firstName
      lastName
    }
    isActive
    createdAt
  }
}
```

#### Update Collection
```graphql
mutation {
  updateCollection(
    id: 8
    name: "Christmas Special Offering"
    isActive: true
  ) {
    id
    name
    isActive
    updatedAt
  }
}
```

#### Delete Collection (Soft Delete)
```graphql
mutation {
  deleteCollection(id: 8)
}
```

#### List All Collections
```graphql
query {
  collections {
    id
    name
    description
    isActive
    createdBy {
      firstName
    }
  }
}
```

## 4. Debtors Dashboard

### Definition

A **debtor** is a parish that **hasn't submitted remittance data for a collection in a given month/year**.

The debtors dashboard tracks:
- Which parishes are missing data
- For which collections
- In which months
- For a specific year

### Database Schema

```sql
CREATE TABLE debtors (
  id SERIAL PRIMARY KEY,
  parish_id INTEGER,           -- Which parish
  collection_id INTEGER,       -- Which collection
  year INTEGER,
  month INTEGER,               -- Which month (1-12)
  expected_amount DECIMAL,     -- Expected amount
  actual_amount DECIMAL,       -- What was actually submitted
  balance DECIMAL,             -- Expected - Actual
  is_paid BOOLEAN,            -- Has this debt been cleared?
  notes TEXT,
  created_at TIMESTAMP,
  updated_at TIMESTAMP,
  UNIQUE(parish_id, collection_id, year, month)
);
```

### Debtors Queries

#### Get All Debtors for a Year (with filters)
```graphql
query {
  debtors(
    year: 2024
    parishId: 1        # Optional
    collectionId: 2    # Optional
    month: 3           # Optional
  ) {
    id
    parish { name }
    collection { name }
    year
    month
    monthName
    expectedAmount
    actualAmount
    balance
    isPaid
    notes
  }
}
```

**Use Cases:**
- Get all debtors for 2024: `debtors(year: 2024)`
- Get debtors for St. Joseph in 2024: `debtors(year: 2024, parishId: 1)`
- Get debtors for First Collection in 2024: `debtors(year: 2024, collectionId: 1)`
- Get debtors for March 2024: `debtors(year: 2024, month: 3)`

#### Get Debtors Summary (Aggregated)
```graphql
query {
  debtorsSummary(
    year: 2024
    parishId: 1        # Optional - Get summary for specific parish
    collectionId: 2    # Optional - Get summary for specific collection
  ) {
    parish { name }
    collection { name }
    year
    totalExpected
    totalActual
    totalBalance
    monthlyBreakdown {
      month
      monthName
      expectedAmount
      actualAmount
      balance
    }
  }
}
```

#### Get Debtors for Specific Parish + Collection
```graphql
query {
  debtorsByParishAndCollection(
    parishId: 1
    collectionId: 2
    year: 2024
  ) {
    parish { name }
    collection { name }
    year
    totalExpected
    totalActual
    totalBalance
    monthlyBreakdown {
      month
      monthName
      expectedAmount
      actualAmount
      balance
    }
  }
}
```

#### Diocese-Wide Debtors Summary
```graphql
query {
  dioceseFinalDebtorsSummary(year: 2024) {
    parish { name }
    collection { name }
    year
    totalExpected
    totalActual
    totalBalance
  }
}
```

### Debtors Mutations

#### Create Debtor Record
```graphql
mutation {
  createDebtor(
    parishId: 1
    collectionId: 2
```

## 5. Authorization Rules

### ADMIN Users
```
✅ Can create parishes
✅ Can create/update/delete collections
✅ Can upload remittance spreadsheets
✅ Can view diocese financial summary
✅ Can view/manage debtors dashboard
✅ Can manage users (create, deactivate)
✅ Can generate/rotate/revoke priest tokens
✅ Can access all parishes
```

### BISHOP Users
```
✅ Can view diocese financial summary
✅ Can view individual parishes (read-only)
✅ Can view debtors dashboard
✅ Can view monthly/yearly remittance totals
❌ Cannot create parishes
❌ Cannot upload data
❌ Cannot manage users
❌ Cannot manage collections
```

### PRIEST Users
```
✅ Can view ONLY their own parish
✅ Can view their parish monthly remittance
✅ Can view their parish yearly remittance
❌ Cannot view other parishes
❌ Cannot upload data
❌ Cannot view diocese-wide views
❌ Cannot access admin features
```

## 6. API Workflow Examples

### Example 1: Admin Setup (First Time)

```graphql
# Step 1: Admin creates parish
mutation {
  createParish(
    name: "St. Joseph Parish"
    diocese: "Lagos Diocese"
    location: "Ikoyi, Lagos"
  ) { id }  # Returns: 1
}

# Step 2: Admin creates collections
mutation {
  createCollection(name: "First Collection") { id }      # Returns: 1
}
mutation {
  createCollection(name: "Second Collection") { id }     # Returns: 2
}

# Step 3: Admin creates priest user
mutation {
  registerUser(
    firstName: "Reverend"
    lastName: "Obinna"
    role: PRIEST
    parishId: 1
  ) { id }  # Returns: 5
}

# Step 4: Admin generates priest token
mutation {
  generatePriestToken(priestUserId: 5) {
    priestToken  # Returns: "a1b2c3d4e5f6..."
  }
}

# Admin shares token with priest (verbally, email, print)
```

### Example 2: Priest Login & View Parish Data

```graphql
# Priest logs in with token
mutation {
  loginWithPriestToken(token: "a1b2c3d4e5f6...") {
    token        # JWT to use in headers
    user { firstName parish { name } }
  }
}

# Set Authorization header to returned token
# Authorization: Bearer <jwt_token>

# Priest views their parish
query {
  myParish {
    name
    diocese
    location
  }
}

# Priest views their yearly remittance
query {
  parishYearlyRemittance(parishId: 1, year: 2024) {
    parish { name }
    yearTotal
    monthlyTotals {
      month
      monthName
      amount
    }
  }
}
```

### Example 3: Admin Tracks Debtors

```graphql
# Get all debtors for 2024
query {
  debtors(year: 2024) {
    parish { name }
    collection { name }
    month
    monthName
    balance  # If balance > 0, they're a debtor
  }
}

# Get summary for St. Joseph parish
query {
  debtorsSummary(
    year: 2024
    parishId: 1
  ) {
    parish { name }
    totalBalance
    monthlyBreakdown {
      month
      monthName
      balance
    }
  }
}

# Record a partial payment
mutation {
  updateDebtor(
    id: 15
    actualAmount: 45000
  ) {
    balance  # Will decrease
  }
}

# Mark as fully paid
mutation {
  markDebtorAsPaid(id: 15) {
    isPaid
    balance  # Will be 0
  }
}
```

### Example 4: Priest Rotation

```graphql
# Old priest leaves, new priest arrives at St. Joseph

# Admin rotates token
mutation {
  rotatePriestToken(priestUserId: 5) {
    priestToken  # New token: "z9y8x7w6v5u4..."
  }
}

# Share new token with new priest
# Old token is now invalid
# Old priest cannot login anymore
```

## 8. Migration Notes (If Upgrading)

If you're upgrading from the old system:

1. **Rename `remittance_sources` to `collections`**
   ```sql
   ALTER TABLE remittance_sources RENAME TO collections;
   ALTER TABLE remittance_sources ADD COLUMN created_by INTEGER;
   ALTER TABLE remittance_sources ADD COLUMN is_active BOOLEAN DEFAULT true;
   ```

2. **Update user roles**
   ```sql
   -- Convert SUPER_ADMIN to ADMIN
   UPDATE users SET role = 'ADMIN' WHERE role = 'SUPER_ADMIN';
   ```

3. **Add priest_token columns**
   ```sql
   ALTER TABLE users ADD COLUMN priest_token VARCHAR(255) UNIQUE;
   ALTER TABLE users ADD COLUMN token_generated_by INTEGER;
   ALTER TABLE users DROP CONSTRAINT email_required_for_non_priest;
   ```

4. **Create debtors table**
   ```sql
   CREATE TABLE debtors (...) -- See schema.sql
   ```

## 9. Security Considerations

### Priest Token Security
- Tokens are 32-character hex strings (256-bit entropy)
- Not passwords - just opaque identifiers
- Cannot be "cracked" - completely random
- Should be treated as sensitive (don't share publicly)
- Admin can revoke anytime

### Best Practices
1. **Admin should securely communicate tokens** (don't email, SMS preferred)
2. **Rotate tokens on priest reassignment** (don't reuse)
3. **Revoke tokens when priests leave** (immediately)
4. **Audit log tracks all token operations** (who, when, what)
